## Bugs

### 1. Redraw artefacts / tearing

**Observation from export:** The screen is redrawn while no real output is
changing, producing tearing and vertical jumps. The report links this to
messages not being rendered when expected, possibly during orchestrator
"missed" messages. Another clue is that the input line/status are redrawn at
higher positions, causing content to jump up and down.

**Hypothesis:** Differential rendering in `tui/compositor.go` is triggering a
full redraw (`needsFullRedrawForChange` / `earlyFullRenderPath`) when the
content has not actually changed, or the viewport top is being recomputed
unstably. The input line and status bar are separate layers/components whose
relative positioning may shift when a full redraw occurs, especially if the
chat viewport height changes between frames.

**Test approach:**
- Write a `Filmstrip` test that replays a stable sequence of orchestrator
events (thinking → content → tool call → tool result) and asserts that
`FullRedrawCount()` does not increase on frames where no visible content
changed.
- Assert that the Editor node's `Rect.Y` and the Status node's `Rect.Y` are
stable across the sequence (input/status do not jump rows).

**Fix approach:**
- Audit `needsFullRedrawForChange` and `earlyFullRenderPath` for conditions
that fire when only non-content layers (footer, status bar) change.
- Ensure the chat viewport height is deterministic when the content height is
unchanged, so the input line layer stays at a fixed Y.
- Ensure the editor's rendered height is also stable when the buffer shrinks,
so the footer below it is not pulled up.

**Status:** Fixed. `tui/chat_viewport.go` tracks `layoutStableHeight` and pads
its rendered slice so the chat viewport never shrinks between terminal resizes.
`tui/editor_render.go` tracks `stableMaxLines` and reserves the largest editor
height seen since the last resize, so the editor itself also stays
monotonically non-decreasing. This removes the compositor's full-clear-on-shrink
path and stops the input/status lines from jumping up.

**Tests:**
- `tui/chat_viewport_height_test.go` — `TestChatViewport_RenderHeightMonotonic`
- `tui/chat_viewport_position_stability_test.go` — `TestChatViewport_PositionStability_InputLineDoesNotJumpUp`
- `tui/editor_height_stability_test.go` — `TestEditor_RenderHeightIsStable`
- `tui/chat_viewport_scrollback_after_shrink_test.go` — `TestChatViewport_ShrinkDoesNotEraseScrollback`

**Observation:** The LLM appeared to stop generating with no activity. The
report wonders whether goa was not consuming tokens and needed a timeout to
retry, or whether there was a starvation/lock.

**Hypothesis:** This is likely a provider-level stall or a goroutine stuck in
a select without a timeout. The TUI/status path may not be signalling the
stall correctly.

**Test approach:**
- Add a regression test that simulates a long pause in the event stream and
asserts the status spinner remains active ("Processing...") and that a
timeout/reconnect path is eventually triggered.
- This may require changes in the provider/orchestrator layer, not the TUI.

**Fix approach:**
- Add a read timeout to the orchestrator's event consumer so a stalled
sub-agent is detected and surfaced as a status message or error.
- Ensure the status spinner is not cleared while waiting for the next chunk.

*Note: Because this may be environmental, the primary fix is observability and
a deterministic timeout, not a UI-only patch.*

### 4. Orchestration tool calls stay open / stuck at bottom — NOT fixed in live use

**Observation:** Tool calls (`orchestrator delegate`, `coder write`) remain stuck at the bottom of the conversation even after completion. The desired behavior is either (a) a FIFO list of open items that move up as new content is appended, or (b) strict chronological order with a visual indicator for active items.

**First attempted fix:** Removed the two-zone active/inactive sort in `tui/chat_viewport.go`; all entries now render in chronological FIFO order. This was the simpler option the user accepted, but it does not distinguish active from completed widgets, so a completed tool widget can sit below a streaming widget if it was created later. The widget-ordering rules are still not fully satisfied.

**Correct direction:** Adopt the simpler chronological order with a clear "active" indicator per widget, and ensure the layout is bottom-anchored so widgets do not jump around.

### 5. No tool calls visible in dedicated mode view (per-agent tab)

**Observation:** In a dedicated mode view (e.g. coder tab), only text is
visible; tool calls are not shown.

**Hypothesis:** The per-agent filter (`ChatViewport.agentFilter`) is applied
in `appendEntry` by checking `Meta["agent"]`. Tool widgets created via
`AddAgentToolExecution` are stamped with `Meta["agent"]`, so they should be
visible. However, the widget's `agentLabel` may not match the disambiguated
label used as the filter, or the tool entry may be created before the filter
is set and not re-rendered.

**Test approach:**
- Reuse `TestOrchestratorViewForwarder_RendersTabbedView` but switch to the
coder tab and assert the `write` tool widget is visible.
- Assert `ChatViewport` entries with `Meta["agent"]` matching the filter are
rendered.

**Fix approach:**
- Ensure `AddAgentToolExecution` stamps the filter key exactly as the
`AgentStreamRegistry` labels it.
- Ensure `SetAgentFilter` invalidates the cache so the per-agent view is
rebuilt.

**Status:** Fixed. `AddAgentToolExecution` was using `LastWhere` which
returns a copy, so the `Meta["agent"]` stamp never persisted. Switched to
`UpdateLast` which modifies the entry in place. Added
`TestChatViewportAgentFilter_IncludesToolWidgets` and
`TestOrchestratorPerAgentTab_ToolWidgetVisible`.

### 6. Scroll does not work (Conversation, coder tabs)

**Observation:** Scrolling creates artefacts and does not fully redraw. The
exported example shows many thinking blocks followed by a content line.

**Hypothesis:** The compositor's differential scroll path
(`emitViewportScroll`) may not handle the chat viewport's internal scrolling
when the user manually scrolls up. The viewport may be bottom-anchored too
aggressively, or the `ChatViewport` render cache is not invalidated when the
scroll offset changes.

**Test approach:**
- Drive a scroll-up key event through the `ChatViewport` in the orchestration
scenario and assert the visible top row changes.
- Assert `FullRedrawCount()` stays low and the visible text is correct after
scroll.

**Fix approach:**
- Add scroll offset tracking to `ChatViewport` and ensure it invalidates the
render cache when the offset changes.
- Ensure the compositor does not force a full redraw on a manual scroll.
- Keep the chat viewport height monotonically non-decreasing so that the
scrollback is not corrupted by a full clear when the content shrinks.

**Status:** Fixed. The scroll approach is unchanged: the terminal's native
scrollback is used. The artefacts were caused by the chat viewport shrinking
and the compositor doing a full screen/scrollback erase. The chat viewport now
pads its rendered height to a stable maximum, and a regression test verifies
that shrinking the chat does not erase the scrollback. A dedicated manual
scroll-up key for the chat viewport is not in scope here.

**Tests:**
- `tui/chat_viewport_scrollback_after_shrink_test.go` — `TestChatViewport_ShrinkDoesNotEraseScrollback`
- `tui/streaming_scroll_test.go` — existing scrollback population tests

### 1. Redraw artefacts / tearing — NOT fixed in live use

**Root cause (first attempt):** `tui/buildScene` stacks children vertically from the top, so
when the chat viewport's content shrinks (e.g. a tool widget collapses, a
thinking block is hidden, or the conversation is cleared), the editor and
status/footer components below it are pulled UP the screen. The compositor's
`clearOnShrink` path then performs a full clear/redraw, producing tearing and
position jumps. The same thing happened when the editor itself shrank after a
multi-line message was submitted/cleared, pulling the footer up.

**First attempted fix:** `tui/chat_viewport.go` tracked `layoutStableHeight` and padded
its rendered slice so the chat viewport never shrinks between terminal resizes.
`tui/editor_render.go` tracked `stableMaxLines` and reserved the largest editor
height seen since the last terminal resize, so the editor never shrank either.
This removed the compositor's full-clear-on-shrink path, but it introduced a
new symptom: the viewport slowly grows downward at every refresh, pushing the
footer down continuously. The stable-height approach is a band-aid that trades
upward jumps for downward drift.

**Correct direction:** The TUI must be bottom-anchored (input/status fixed at the
bottom, conversation scrolls above it) instead of top-down with padding.

**Tests:**
- `tui/chat_viewport_height_test.go` — `TestChatViewport_RenderHeightMonotonic`
- `tui/chat_viewport_position_stability_test.go` — `TestChatViewport_PositionStability_InputLineDoesNotJumpUp`
- `tui/editor_height_stability_test.go` — `TestEditor_RenderHeightIsStable`
- `tui/chat_viewport_scrollback_after_shrink_test.go` — `TestChatViewport_ShrinkDoesNotEraseScrollback`

## 2026-07-08 regression: previous TUI fixes did not work in live use

After the first round of fixes, a live hub-topology run showed the UI is still
broken:

- The orchestrator spinner is stuck ("orchestrator answering..." does not update).
- The conversation content slowly moves down at each refresh.
- The widget-ordering rules are still not correctly implemented.

These new issues are now logged in `bugs.md` and are the subject of the
critical review below.

### Critical review of the TUI layout approach

The current layout model is **not sound** for this use case. The fixes applied
in the first round (stable-height padding and chronological FIFO ordering) are
band-aids that introduce new problems:

1. **Top-down stacking is wrong.** `tui/tui.go` `buildScene` stacks components
from top to bottom. Any change in the conversation/agent-content height
shifts the input and status lines up or down. A chat UI should be anchored at
the bottom: input and status stay fixed, and the conversation area scrolls
above them.

2. **Stable-height padding causes downward drift.** `ChatViewport.padToStableHeight`
prevents the viewport from shrinking, but it also grows monotonically whenever
the raw content height exceeds the previous value. The status spinner invalidates
running tool widgets on every frame, forcing repeated rebuilds. Any height
jitter in those rebuilds causes the viewport to grow and push the footer down
continuously.

3. **Terminal scrollback is fragile.** The compositor relies on emitting newlines
into the terminal scrollback. This is coupled with the top-down layout and the
differential renderer. If any component changes height, the scroll math can
leave blank lines or corrupt the scrollback. There is no in-app scroll-up key,
so the user must depend on the terminal's scrollbar.

4. **Status spinner rebuilds the whole chat viewport every frame.**
`StatusMsg.SetOnFrameChange` -> `ChatViewport.InvalidateRunningToolWidgets` ->
full `ChatViewport` rebuild on every spinner tick. This is expensive and can
interact with the stable-height padding. The spinner should only invalidate the
tool widgets that need to be redrawn, not the whole conversation.

5. **Multi-pane conversation adds unnecessary complexity.** The global
conversation + per-agent tabs + stats panel share a single `ChatViewport` with
suppression and filtering. This makes position stability and scroll correctness
harder because the same viewport must maintain stable height across different
content sets. The simpler fallback (global conversation + footer stats) is
preferable.

### Recommended direction

- **Option A (full fix):** Redesign the layout as bottom-anchored. Input/status
are fixed at the bottom. The conversation area fills the remaining space above,
clamped to available height. When the conversation grows beyond its allocated
height, the oldest lines scroll into the terminal scrollback, but the visible
region stays at the same Y. Add a manual scroll-up key for the chat viewport.

- **Option B (fallback):** If the full fix is too large, keep only the global
conversation and add a compact stats strip above the input line. Remove the
per-agent tabs and the stats panel. This is the fallback the user explicitly
accepts.
