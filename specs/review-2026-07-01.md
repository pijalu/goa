# Codebase Review & Execution Plan — UX Input Discipline + Clarification Tool

Branch: **`feature/review`** (to be created from `main`)
Date: 2026-07-01
Status: **Plan — ready for automatic execution**

---

## 1. Review Findings

### 1.1 UX — User text input is NOT routed through the main input zone

**Root design smell.** There are **two parallel input mechanisms** that overlap in
purpose, and the wrong one is used in the most common interactive flows:

| Mechanism | Where typing happens | Reuses main editor (history/kill-yank/autocomplete/undo/title)? |
|---|---|---|
| `TUI.ShowInput(prompt, current)` → `inputOverlay` | A **separate** `Input` overlay component that captures input | **No** — second input region |
| `App.requestMainInput(prompt, onSubmit)` → `pendingInput` + `Editor.SetTitle` | The **main** input line editor | **Yes** ✅ |

The established, correct pattern (`requestMainInput`) is used by `/goal`,
`/export`, and the review comment/confirm handlers (`internal/app/events.go`).
The incorrect pattern (`ShowInput`) is used by the config/model/workflow menus:

- `core/commands/config.go:662` — **API key for a preset provider** ← the reported case
- `core/commands/config.go:673` — custom provider endpoint
- `core/commands/config.go:679` — provider API key (custom)
- `core/commands/config.go:693` — provider ID
- `core/commands/config.go:737` — add a model for a provider
- `core/commands/config.go:865` — model temperature
- `core/commands/model.go:205` — custom model name
- `core/commands/workflows.go:262` — describe feature to implement

Because `ShowInput` spawns a throwaway overlay, the user loses editor history,
completion, and the live title cue that `requestMainInput` provides.

**Missing artifact:** there is **no documented UI guideline** stating that *all*
user text input must flow through the main input zone with a title. Without it,
both patterns look equally valid and new code keeps picking the wrong one.

### 1.2 Missing element — `tools/ask` clarification tool is dead code

`tools/ask/ask_user.go` defines `AskUserQuestionTool`, but:

- The package `github.com/pijalu/goa/tools/ask` is **imported nowhere**
  (`grep` confirms zero importers outside the package itself).
- It is **never registered** in `internal/app/bootstrap.go:registerTools` nor
  `internal/app/subsystems.go`.
- Its `Ask func(...)` field has **no wiring** to the TUI; the only fallback is
  `defaultAsk`, which always returns `"no question handler configured"`.

**Consequence:** the LLM has **no** way to ask the user a clarifying question.
This is the core missing capability requested.

### 1.3 Missing config gate — no default-on clarification toggle

`ToolEnabledConfig` (`config/config.go:301`) has flags only for opt-IN tools
(`BGExec`, `DelegateTo`, `Memento`, …), all defaulting to `false`. The required
behavior is the inverse: **enabled by default, disableable by config**. No such
flag exists.

### 1.4 Bugs / minor design issues

- **Tautology** in `internal/app/bootstrap.go` (~line 472):
  `if cfg.Tools.SmartSearch.Enabled || !cfg.Tools.SmartSearch.Enabled {` is
  always true. It should be an unconditional registration (or a real toggle).
- **Stale docs** — `tools/ask/ask_user.short.md` / `.long.md` document an
  unwired tool with a schema that no longer matches the desired richer design.
- **`ShowInput` contract ambiguity** — `core.Context.ShowInput` returns
  `(value, ok)` where `ok` is derived from `value != ""`, conflating "user
  cancelled" with "user submitted empty". `requestMainInput` treats empty submit
  as cancel explicitly, which is clearer.

### 1.5 Build — Windows cross-compile is broken

The CI Windows build (`GOOS=windows GOARCH=amd64 CGO_ENABLED=0 go build
./cmd/goa`) fails:

```
internal/sandbox/kill_windows.go:17: undefined: syscall.GenerateConsoleCtrlEvent
internal/sandbox/runner_windows.go:14: undefined: syscall.CREATE_NO_WINDOW
```

**Root cause:** `syscall.GenerateConsoleCtrlEvent`, `syscall.CTRL_BREAK_EVENT`,
and `syscall.CREATE_NO_WINDOW` do **not** exist in Go's standard `syscall`
package on Windows. They live in `golang.org/x/sys/windows`. The code was
written against the wrong package.

**Evidence the fix is trivial & consistent with the codebase:**
- `golang.org/x/sys v0.46.0` is already a direct dependency (`go.mod`).
- `tui/terminal_windows.go` already imports `golang.org/x/sys/windows` and
  uses it correctly (`windows.GetConsoleMode`, etc.).
- The two broken files (`internal/sandbox/kill_windows.go`,
  `internal/sandbox/runner_windows.go`) are the only outliers still using
  `syscall` for these symbols.

This is the failure behind the `windows-amd64` matrix job in
`.github/workflows/build.yml` (Step `Build`, `GOOS=windows`). The same gate
must be run locally before push so it is caught early (see Milestone 6b).

### 1.6 Repo layout — landing page (`index.html`) lives in the wrong place

A 16 KB `index.html` project landing page is tracked directly in the
repository **root**. It does not belong there:

- It clutters the source root and is unrelated to the Go module.
- It is never built/served anywhere (there is no GitHub Pages workflow — only
  `.github/workflows/build.yml`, which builds binaries).
- Project landing pages belong on **GitHub Pages**, served from a `/docs`
  folder, a dedicated `gh-pages` branch, or a separate `web/`/`site/`
  directory — not the source tree root.

## 2. UI Guideline (to be added to `docs/TUI.md`)

> ### Input discipline
> **All user text input must flow through the main input zone** (the persistent
> `Editor`), never through a throwaway overlay `Input`.
>
> 1. Hosts capture input via `requestMainInput(prompt, onSubmit)` (or the
>    `core.Context.RequestMainInput` callback), which sets the editor **title**
>    to the prompt so the user always knows what is being asked.
> 2. A **modal/card rendered inside the conversation viewport** may accompany the
>    prompt to display richer context (title, summary, options), but the *typing*
>    still happens in the main editor.
> 3. The legacy `ShowInput`/`ShowInputFunc` overlay path is **deprecated** for
>    free-text capture and retained only as a transition shim; new code must not
>    use it for user input.

### Milestone 3 — Host callback to ask via the main input zone

**Files:** `core/context.go`, `internal/app/commandcontext.go`

3.1 Add to `core.Context`:
```go
// ClarifyFunc shows a clarification card in the conversation and routes the
// answer through the main input line. It blocks until the user answers or
// cancels. ok==false means the user cancelled.
ClarifyFunc func(card *tui.ClarifyCard) (answer string, ok bool)
```
   (Import `tui` is already present in `core/context.go` via `tui.SelectorItem`.)

3.2 Add the convenience method `func (c Context) Clarify(...)` mirroring the
    existing `ShowInput` helper style.

3.3 Wire in `internal/app/commandcontext.go`:
```go
ClarifyFunc: func(card *tui.ClarifyCard) (string, bool) {
    if subs.chat != nil { subs.chat.AddClarifyCard(card) }
    ch := make(chan clarResult, 1)
    app.requestMainInputWithCancel(card.title+": "+card.question, func(text string) {
        ch <- clarResult{text, true}
    }, func() { ch <- clarResult{"", false} }, true)
    r := <-ch
    return r.text, r.ok
},
```
   (Use a tiny local `type clarResult struct{ text string; ok bool }`.)
   This reuses the **proven** `requestMainInputWithCancel` path (title on the
   editor, cancel via Ctrl+C/empty submit), so no new input plumbing is needed.

**Tests:** covered end-to-end through the tool test (Milestone 4) with a stubbed
`ClarifyFunc`; plus a focused test in `core` that `Clarify()` dispatches to
`ClarifyFunc` and returns its values.

### Milestone 4 — The clarification tool (series of questions)

**Files:** `tools/ask/ask_user.go` (rewrite), `tools/ask/ask_user_test.go`,
new embed docs `tools/ask/ask_user.short.md`, `tools/ask/ask_user.long.md`,
new renderer `tools/ask/ask_user_renderer.go` + test.

4.1 New schema — the model sends a **series** of questions in one call; the tool
     asks each **separately** and returns the aggregated questionnaire:
```go
type clarifyQuestion struct {
    Title        string   `json:"title"`
    Summary      string   `json:"summary"`
    Question     string   `json:"question"`
    Options      []string `json:"options,omitempty"`
    Required     bool     `json:"required,omitempty"`
    AllowFreeText bool    `json:"allow_free_text,omitempty"` // default true
}
type clarifyInput struct {
    Questions []clarifyQuestion `json:"questions"`
}
```
Tool name: **`ask_user_question`** (keep existing name so docs/prompt refs stay
valid). Description (visible to the model):
> "Ask the user one or more clarifying questions when requirements are
> ambiguous. Each question is shown as a card (title/summary/question/options)
> and answered through the main input line. Use this BEFORE starting uncertain
> work. Disabled when `tools.enabled.clarify_disabled` is true."

4.2 `Execute` loop — for each question:
   a. Validate (`Question` required; `Options` capped at e.g. 6).
   b. Build a `*tui.ClarifyCard`.
   c. Call `host.ClarifyFunc(card)` (blocking). If `!ok` and `Required` ⇒ return
      a `ToolError` of type `cancelled`; if `!ok` and not required ⇒ record
      empty/skipped.
   d. If `Options` non-empty and answer matches an option number/label, store the
      canonical option text; otherwise store the raw free-text answer.
   Aggregate into:
```go
type clarifyAnswer struct {
    Question string `json:"question"`
    Answer   string `json:"answer"`
    Skipped  bool   `json:"skipped,omitempty"`
}
```
   Return `fmt.Sprintf("[ask_user_question] %d question(s) answered:\n%s", n, json)`.
   `Execute` runs synchronously on the agent loop, so blocking is fine (same as
   the existing `Ask func` contract).

4.3 Host dependency injection — the tool gets `ClarifyFunc` via a field set at
   registration time (mirrors `AskUserQuestionTool.Ask`):
```go
type AskUserQuestionTool struct {
    agentic.BaseTool
    Clarify func(*tui.ClarifyCard) (string, bool)
}
```
   Keep a `defaultClarify` that returns `("", false)` + error when unwired, so
   unit tests without a host behave deterministically.

4.4 Renderer (`tools/ask/ask_user_renderer.go`) — registers via
   `tui/register_renderers.go` following the existing renderer pattern; shows a
   compact "❓ asked N questions" line that expands to the Q/A pairs.

4.5 Docs — rewrite `ask_user.short.md` / `ask_user.long.md` to the new schema
   and add the clarification usage guidance.

**Tests** (`tools/ask/ask_user_test.go`, table-driven):
- `TestClarify_Series` — 3 questions, stubbed host returns canned answers ⇒
  returned JSON contains all three answers in order.
- `TestClarify_RequiredCancelled` — user cancels a required question ⇒
  `ToolError` type `cancelled`.
- `TestClarify_OptionCanonicalization` — answer `"2"` maps to `options[1]`.
- `TestClarify_Validation_MissingQuestion` ⇒ `ToolError` `missing_question`.
- `TestClarify_DefaultHostError` — unwired tool ⇒ error.
- Renderer test: `TestAskUserRenderer_Render`.

### Milestone 6b — Fix Windows cross-compile build

**Files:** `internal/sandbox/kill_windows.go`,
`internal/sandbox/runner_windows.go`.

These files use symbols from the standard `syscall` package that do not exist
on Windows; switch to the already-vendored `golang.org/x/sys/windows`, matching
the established pattern in `tui/terminal_windows.go`.

6b.1 `internal/sandbox/kill_windows.go` — replace import + call:
```go
import (
    "os/exec"
    "golang.org/x/sys/windows"
)
func killProcessGroup(cmd *exec.Cmd) {
    if cmd.Process == nil { return }
    _ = cmd.Process.Kill()
    _ = windows.GenerateConsoleCtrlEvent(windows.CTRL_BREAK_EVENT,
        uint32(cmd.Process.Pid))
}
```
6b.2 `internal/sandbox/runner_windows.go` — `cmd.SysProcAttr` is typed
   `*syscall.SysProcAttr` (stdlib), but its `CreationFlags` field is a plain
   `uint32`, so the `windows.CREATE_NO_WINDOW` constant value assigns into it
   cleanly:
```go
import (
    "os/exec"
    "syscall"
    "golang.org/x/sys/windows"
)
func applyPreexec(cmd *exec.Cmd, fn func() error) {
    cmd.SysProcAttr = &syscall.SysProcAttr{
        CreationFlags: windows.CREATE_NO_WINDOW,
    }
    _ = fn
}
```

**Tests / validation:**
- Cross-compile gate (add to CI if missing): `GOOS=windows GOARCH=amd64
  CGO_ENABLED=0 go build ./cmd/goa`, plus a `GOOS=linux` regression guard.
- Because a Windows host is required to *run* the tests, the cross-compile is
  the acceptance gate; `go vet ./...` on the dev OS will **not** surface this
  — only `GOOS=windows go build` does.

## 4. Acceptance Criteria

1. Adding a provider captures the API key via the **main input zone** with a
   title — verified by pty test.
2. A documented "Input discipline" guideline exists in `docs/TUI.md`.
3. The model has an `ask_user_question` tool that:
   - accepts a series of `{title, summary, question, options}`,
   - renders a card in the conversation for each,
   - collects each answer from the main input line,
   - returns the aggregated answers to the model.
4. The tool is **registered by default** and disappears when
   `tools.enabled.clarify_disabled: true`.
5. The tool's schema description explicitly tells the model it is for requesting
   clarification.
6. All quality-gate commands pass with no new violations.

## 5. Risk / Notes

- `Execute` blocks per question. The agent loop already expects blocking tools
  (existing `Ask` contract), and `requestMainInputWithCancel` integrates with
  Ctrl+C cancellation, so a stuck question is recoverable.
- The `tools/ask` package currently has **zero importers**; rewriting it is safe
  (no callers to break). Its tests are the only consumers.
- Migrating `ShowInput` (Milestone 6) is the highest-touch change; do it last so
  the new tool ships even if migration is deferred.
